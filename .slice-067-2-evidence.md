# Slice 067-2 — Gate config out-of-band + read-only (evidence)

Branch: `slice/067-2-gate-config-oob` off `slice/067-3-approve-resim` @ `62f569e0`.

## The hole this slice closes
The worker-time convergence gate runs `npm run check-candidate -- <taskDir> <candidate>`
(via `subprocessGauntlet`, src/create/gauntlet.ts). `check-candidate.ts main()` called
`loadGateConfig(taskDir)`, which read `<taskDir>/gate.json` and let "that config win".
In the server era a submitter ships the whole task dir, gate.json included, so a malicious
`{"bandMin":0,"bandMax":1,"floor":0}` would wave any garbage candidate through. The bar must
come from the trusted driver, not the untrusted task.

## What changed
- `src/check-candidate.ts`
  - `loadGateConfig(taskDir, trustTaskGate = false)` — **default flips to out-of-band**:
    when `trustTaskGate` is false (the authoritative, untrusted path) the task dir's
    gate.json is IGNORED and `defaultGateConfig()` (trusted tunables) wins. When true
    (explicit opt-in) the old merge-over-defaults behaviour returns.
  - `main()`: parses `--trust-task-gate` (the only flag), strips it from the positionals,
    passes it to `loadGateConfig`. USAGE + header doc updated.
- `src/check-candidate.test.ts` — replaced the "gate.json is read" test with:
  default-ignores-gate.json, permissive-gate-ignored, `--trust-task-gate` honours override,
  trusted-no-gate falls back; plus a new `untrusted-submission threat model` block showing
  PRE (trust → overtuned waved through) vs POST (default → overtuned bounced).
- `tasks/frostbite-striker/README.md` — documents that the authoritative bar is the trusted
  defaults; the shipped gate.json (which mirrors the defaults) only overrides under `--trust-task-gate`.

The trusted override flag lives in `check-candidate.ts main()` (`--trust-task-gate`) and the
function signature (`loadGateConfig(taskDir, trustTaskGate)`).

## Permissive-gate.json repro (real CLI, end-to-end)
Task dir = copy of frostbite-striker with `gate.json = {"bandMin":0,"bandMax":1,"floor":0}`,
out/candidate.json = the overtuned 999-power fixture.

PRE-slice (honour task gate via `--trust-task-gate`):
```
{"status":"passed","verdict":"in-band","overallWinRate":1,"band":{"min":0,"max":1},"floor":0}
exit 0   ← garbage waved through
```
POST-slice (default — permissive gate.json ignored):
```
{"status":"gate-bounced","verdict":"overtuned","overallWinRate":1,"band":{"min":0.35,"max":0.65},"floor":0.25}
process exit 2   ← bounced against the trusted band
```

## Worker-time meta is trusted-source — confirmed
`check-candidate.ts` imports `REFERENCE_META` (src/content/reference-meta.ts) and
`stressRegistry` (src/content/stress.js) as modules; `checkCandidate` calls
`runGate(candidate, REFERENCE_META, config, stressRegistry)`. No file is read from the
task dir for the meta or the registry. The gauntlet only passes `taskDir` + candidate path.
So nothing in the task dir can swing which meta the convergence gate sims against — same
anchoring slice 3 established for `approve`.

## Adversarial review of my own work — can the task dir still swing the bar?
- The default path (`trustTaskGate=false`) never opens or reads gate.json — short-circuits
  before the `readFileSync`. A planted gate.json (any contents, including non-JSON) has zero
  effect; the malformed-JSON throw is unreachable on the default path.
- The worker-time gauntlet (`subprocessGauntlet`) spawns `check-candidate` with NO
  `--trust-task-gate`, so worker-time convergence is on the safe default.
- Reference meta / registry are module imports, not task-dir reads.
- Residual: a CALLER that passes `--trust-task-gate` against an untrusted task dir reopens the
  hole — but that is the explicit, named opt-in (local dev). No production/worker path passes it.
  `grep -rn "trust-task-gate\|trustTaskGate" src` shows it only in check-candidate.ts and its test.
